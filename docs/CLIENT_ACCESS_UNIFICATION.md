# Унификация клиентского доступа: перевод web-frontend на gateway

## Текущая ситуация
- **web-frontend** (порт 3000) – пользовательский веб-интерфейс, доступный из интернета.
- Сейчас он напрямую обращается к **orchestrator** через gRPC, используя клиентскую библиотеку `internal/search`. Это нарушает принцип единой точки входа и дублирует логику, уже реализованную в gateway (валидация, rate limiting, трассировка, метрики).
- **Gateway** (порт 8443) уже обрабатывает запросы от внешних клиентов (IRC, Telegram) и предоставляет HTTP API для поиска и скачивания.

## Цель изменений
Перевести все запросы от web-frontend на gateway, чтобы:
- Централизовать валидацию, rate limiting, сбор метрик и трассировку.
- Упростить архитектуру: убрать прямое gRPC-соединение между web-frontend и orchestrator.
- Уменьшить дублирование кода (клиент `internal/search` станет не нужен для web-frontend).
- Обеспечить единообразную обработку ошибок и логирование.

## План реализации

### 1. Изменения в web-frontend (`cmd/web-frontend/`)
- Заменить текущие вызовы `searchSvc.SearchWithPagination` (которые используют `internal/search`) на HTTP-запросы к gateway.
- Использовать эндпоинт gateway `/search` (POST) с телом:
  ```json
  {
    "query": "...",
    "page": N,
    "limit": M
  }


    Для скачивания файлов заменить прямые вызовы к downloader на использование gateway: эндпоинт /download/{token} (уже существует).
    Убрать зависимость от internal/search (удалить импорт и, при необходимости, сам пакет из go.mod, если он больше нигде не используется).

2. Изменения в конфигурации (ebusta.yaml)
    Добавить в секцию web_frontend поле gateway_url (например, http://gateway:8443). Это позволит настраивать адрес gateway независимо.
    Удалить устаревшие поля (например, прямые адреса orchestrator), если они были.

3. Изменения в gateway
    Не требуются, так как эндпоинты /search и /download/{token} уже реализованы и работают.

4. Изменения в других сервисах
    orchestrator и остальные сервисы остаются без изменений.

5. Обновление тестов
    Исправить e2e-тесты, которые напрямую обращались к orchestrator, переведя их на gateway.
    Убедиться, что все тесты проходят.

Преимущества после реализации
    Все внешние запросы (веб, IRC, Telegram) будут проходить через единый gateway.
    Упростится добавление новых политик (например, аутентификации) – достаточно добавить их в gateway.
    Улучшится наблюдаемость: метрики и трассировка будут собираться в одной точке.
    Уменьшится сетевая сложность: вместо двух соединений (web-frontend → orchestrator и web-frontend → downloader) будет одно (web-frontend → gateway).

Возможные риски
    Увеличение задержки на один сетевой hop (но это компенсируется централизацией и упрощением).
    Необходимость аккуратно переписать логику пагинации и обработки ошибок на HTTP-клиент (но gateway уже возвращает структурированные ответы).
    Временное дублирование кода, пока старый клиент не будет полностью удалён.

Ожидаемый результат

После завершения задачи web-frontend будет обращаться только к gateway, а прямое gRPC-соединение с orchestrator исчезнет. Архитектура станет более чистой и соответствующей принципу единой точки входа.
