# Ebusta — AI Rules (Codex/Claude)

## Роль
Вы — исполнитель. Архитектура и решения фиксируются человеком (Serge) и главным архитектором (ChatGPT).  
Ваша задача — реализовывать строго то, что описано в задаче, без отклонений.

## Источники истины
1) Исходный код репозитория (Go, Scala, proto, конфиги).
2) Proto-контракты в `api/proto/v1/` (неизменяемы для версии v1).
3) Документация в `docs/` (эти файлы).
4) Файл конфигурации `ebusta.yaml` и его секции.
Если есть конфликт — остановиться и запросить уточнение у архитектора.

## Запрещено
- Добавлять новые технологии/сервисы/слои без явного требования.
- Перепридумывать API, DSL, форматы хранения, коды ошибок.
- Проводить массовые рефакторинги вне задачи.
- Менять сигнатуры существующих gRPC-методов или нумерацию полей proto.
- Нарушать границы компонентов (например, писать в storage из gateway).

## Обязательно
- Делать изменения минимальными по диффу.
- Соблюдать контракты:
  - TraceID обязателен во всех сервисах (передаётся через `X-Trace-Id` / `x-trace-id`).
  - Ошибки должны соответствовать маппингу `docs/API_ERROR_MAPPING.md`.
  - gRPC-методы должны использовать общие типы ошибок (`errutil.AppError`).
- В конце ответа всегда давать:
  1) Что сделано (3–7 пунктов).
  2) Какие файлы изменены.
  3) Как запустить/проверить (команды).
  4) Риски/заметки (если есть).

## Формат результата
- Предпочтительно: один коммит на задачу.
- Коммит-сообщение: "component: кратко что сделано" (например, "gateway: add trace propagation to download handler").
